It is our view that RMI sucks. Admittedly it sucks less in 2008 than it did ten years ago, but it is still preoccupied on forcing end users to be aware of the eight fallacies of distributed computing, for every invocation. From the outset, these are what were annoying about RMI :

  • RMI forces you to extend 'UnicastRemoteObject' -fixed in latter JDKs
  • RMI forces you to catch a checked exception - 'RemoteException' - wherever methods are called.
  • RMI forces interfaces to implement 'Remote'
  • Cumbersome RMI stub compiler - fixed in latter JDKs
  • Static, though optional, RMI registry

A developer coding for a remote divide should do so in a way that respects the Facade Pattern but should be able to leave outage concerns to more centralized places.

Centralized places could be, a handler servlet (through which all requests go), or a container such as Spring, PicoContainer or Guice (we all adhere to Inversion of Control these days right?).
